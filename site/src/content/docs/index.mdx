<div class="feature-band">
  <div class="feature-band__body">
    <p class="feature-band__layer">Layer 1 · the atom</p>
    <h2 class="feature-band__title">Resolve a citation</h2>
    <p class="feature-band__audience">If you just need DOI / PMID / arXiv metadata at the shell.</p>
    <p>
      Hand quartobot a persistent-identifier key — `doi:`, `pmid:`,
      `arxiv:`, `isbn:`, `url:`, `wikidata:`, `pmc:` — and get back
      CSL JSON on stdout. Pipes cleanly through `jq` or any agent's
      shell tool. The resolver is `manubot.cite.citekey_to_csl_item`
      underneath; eight years of source-API quirks behind one call.
    </p>
    <p class="feature-band__cta">
      <a href="./resolve-single-citation/">How to resolve a single citation →</a>
    </p>
    <p class="feature-band__stop">
      Done — you have a single-key metadata lookup tool. Stop here
      or stack on Layer 2.
    </p>
  </div>
  <div class="feature-band__code">

```bash
quartobot resolve --output - doi:10.21105/joss.01686 \
  | jq '.[0].title'
# → "Welcome to the Tidyverse"
```

  </div>
</div>

<div class="feature-band feature-band--reverse">
  <div class="feature-band__body">
    <p class="feature-band__layer">Layer 2 · the manuscript</p>
    <h2 class="feature-band__title">Resolve in the render loop</h2>
    <p class="feature-band__audience">If you author Quarto manuscripts, books, or websites.</p>
    <p>
      One line in `_quarto.yml` makes `quartobot resolve` run before
      pandoc on every render. Authors write `@doi:…` directly in
      prose; the hook fetches each key, writes the CSL JSON to
      `references.json`, and pandoc-citeproc formats the rendered
      output. Network only on first render; cached after.
    </p>
    <p class="feature-band__cta">
      <a href="./first-manuscript/">First manuscript in 15 minutes →</a>
    </p>
    <p class="feature-band__stop">
      Done — your manuscript builds itself with resolved citations.
      Stop here or stack on Layer 3.
    </p>
  </div>
  <div class="feature-band__code">

```yaml
# _quarto.yml
project:
  pre-render: >-
    quartobot resolve --from-scan .
    --output references.json
    --id-mode citation-key

bibliography:
  - references.bib
  - references.json
```

<div class="feature-band">
  <div class="feature-band__body">
    <p class="feature-band__layer">Layer 3 · the agent</p>
    <h2 class="feature-band__title">Ground AI-assisted drafting</h2>
    <p class="feature-band__audience">If you draft with Claude Desktop, Codex, Gemini, Cursor, or any agent harness with a shell.</p>
    <p>
      Two surfaces for agentic citation work — MCP and a stdout CLI —
      both reaching the same resolver. Agents use either to look up
      the citations they propose against the source registrars
      before those cites reach your draft. `quartobot mcp` exposes
      `resolve_citation`, `scan_project`, and `validate_project` as
      MCP tools for native MCP clients. For agents that already live
      in a shell, `quartobot resolve --output -` is the same call
      piped through stdout — equally good, sometimes preferred for
      portability and composition with `jq`.
    </p>
    <div class="feature-band__callout">
      <strong>Same call, same answer.</strong> An agent that routes
      its proposed cite keys through quartobot gets back
      source-registrar metadata — Crossref for DOIs, PubMed for
      PMIDs, arXiv for preprints — using the same resolver your
      manuscript renders with. Agent metadata and rendered metadata
      can't drift apart.
    </div>
    <p class="feature-band__cta">
      <a href="./mcp-claude-desktop/">MCP + Claude Desktop →</a>
      &nbsp;·&nbsp;
      <a href="./shell-tool-agent/">Shell-tool agents →</a>
    </p>
    <p class="feature-band__stop">
      Done — your AI tools cite from the same registrar your
      manuscript does. Stop here or stack on Layer 4.
    </p>
  </div>
  <div class="feature-band__code">

```json
// ~/Library/Application Support/Claude/
//   claude_desktop_config.json
{
  "mcpServers": {
    "quartobot": {
      "command": "quartobot",
      "args": ["mcp"]
    }
  }
}
```

<div class="feature-band feature-band--reverse">
  <div class="feature-band__body">
    <p class="feature-band__layer">Layer 4 · the publication</p>
    <h2 class="feature-band__title">Every commit, a reviewable manuscript artifact</h2>
    <p class="feature-band__audience">If you want per-commit permalinks, PR previews, and "manuscript-as-software" CI.</p>
    <p>
      `quartobot use github-ci` scaffolds the render-and-deploy
      workflow. Each push to your repo renders the manuscript and
      lands it on GitHub Pages; each commit gets an immutable
      `/v/<sha>/` permalink alongside the latest; each PR gets a
      sticky preview-render comment. Reviewers see what merges
      before it merges.
    </p>
    <div class="feature-band__callout">
      <strong>Reviewers see exactly what your branch renders to.</strong>
      The sticky PR comment links to a live preview of the
      manuscript — no local checkout, no "but it works on my
      machine," no version drift between what reviewers read and
      what merges. The `/v/<sha>/` permalink survives forever, so
      anyone can cite a specific version of the paper.
    </div>
    <p class="feature-band__cta">
      <a href="./first-manuscript/#7-add-the-publish-on-every-commit-ci">First manuscript, step 7 →</a>
    </p>
    <p class="feature-band__stop">
      Done — every commit gets a permanent URL, every PR gets a
      preview. Skip this layer entirely if you already have CI or
      use `quarto publish`.
    </p>
  </div>
  <div class="feature-band__code">

```bash
quartobot use github-ci
# scaffolds:
#   .github/workflows/render.yml
#   .github/workflows/pr-closed.yml
#   _version-banner.html
# then prints a YAML snippet to paste
# into _quarto.yml for the banner include
```
